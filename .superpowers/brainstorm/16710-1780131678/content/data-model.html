Modèle de données — Dual-stream décorrélé

Les événements et les screenshots sont deux flux indépendants. L'association se fait dans l'éditeur.

A

Couplage fort à la capture

Chaque screenshot embarque directement les événements qui l'ont déclenché. L'association est figée au moment de la capture.

{
  "id": "step-1",
  "screenshot": "0001.png",
  "timestamp": 1700000000000,
  "events": [
    { "type": "click", "x": 120, "y": 45, "t": 1700000000050 },
    { "type": "key",   "key": "Tab",      "t": 1700000000200 }
  ]
}

Pour

  • Simple à produire
  • Pas de rebinding nécessaire

Contre

  • Frontières d'étapes figées
  • Impossible de déplacer un événement
  • Changement de fenêtre = ambiguïté non résolvable
B

Dual-stream décorrélé ⭐ Recommandé

Deux fichiers séparés. Les événements référencent le screenshot courant par ID, mais l'éditeur peut rebinder librement. Sur changement d'interface, l'auteur décide où placer la frontière.

// screenshots.json
[{ "id":"s1", "file":"0001.png", "t":1000 },
 { "id":"s2", "file":"0002.png", "t":2100 }]

// events.json
[{ "type":"click", "x":120,"y":45, "t":1050,
   "step_id":"s1" },         ← rebindable
 { "type":"window_change",
   "to":"Finder", "t":2000,
   "step_id":"s1" },         ← ou s2, choix éditeur
 { "type":"key","key":"↵",  "t":2050,
   "step_id":"s2" }]

Pour

  • Frontières d'étapes modifiables
  • Changement de fenêtre : ambiguïté résolvable
  • Ajout de screenshots sans recapturer les events
  • Player sait exactement quoi rejouer et dans quel ordre

Contre

  • Éditeur doit exposer le rebinding
  • Légèrement plus complexe
