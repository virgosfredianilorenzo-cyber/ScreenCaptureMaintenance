Problématique de maintenance — quand le logiciel évolue

Scénarios fréquents et ce que le dual-stream permet

Scénario 1 — Reskin visuel (même workflow)

Boutons redessinés, couleurs changées, logo mis à jour. Le clic est au même endroit.

Aujourd'hui : recapturer 100 % de la session
Avec dual-stream : recapturer uniquement les screenshots des étapes concernées — les événements ne changent pas

Scénario 2 — Bouton déplacé ou renommé

« Fichier → Nouveau projet » est maintenant dans une barre latérale à (200, 80) au lieu de (120, 45).

Aujourd'hui : recapturer tout depuis cette étape
Avec dual-stream : éditer l'événement click (coordonnées ou séquence) + recapturer le screenshot de l'étape

Scénario 3 — Étape supplémentaire ajoutée

Une boîte de confirmation a été ajoutée entre l'étape 3 et 4.

Aujourd'hui : recapturer tout depuis cette étape
Avec dual-stream : enregistrer une seule nouvelle étape + l'insérer entre les étapes 3 et 4

⭐ Idée supplémentaire — Détection de changement

Quand le formateur ouvre une session existante, l'app peut comparer automatiquement les screenshots actuels avec les anciens et signaler les étapes qui ont changé.

Flux de maintenance proposé :
1. Rejouer la session sur le nouveau logiciel
→
2. Diff automatique screenshot-par-screenshot
→
3. Étapes "à valider" surlignées
→
4. Formateur valide ou re-capturé step par step

Ce que ça implique pour l'architecture

  • Chaque étape est un objet indépendant avec un ID stable
  • Screenshots versionnés par étape (step-3-v1.png → step-3-v2.png)
  • Re-record d'une étape = nouvelle capture qui remplace l'ancienne sans toucher les voisines
  • Le player utilise toujours les IDs d'étapes, jamais les noms de fichiers
  • Métadonnée status: current | needs_review sur chaque étape
